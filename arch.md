The Next.js frontend (autogpt_platform/frontend) becomes a Render web service, built and started with npm run build / npm start. The rest_server (the main backend API, built from autogpt_platform/backend/Dockerfile) becomes a second Render web service, since it needs a public HTTPS endpoint for the frontend to call. The websocket_server and scheduler_server map to either additional web services (if they need inbound traffic) or private services if they're only reached internally by other backend components. Supabase (currently handling both Postgres and auth) splits into two pieces: the database itself becomes a Render Postgres instance, while the auth logic Supabase provided needs to be reimplemented in the backend code, since Render doesn't offer a managed auth product. Redis maps directly to a Render Key Value instance. ClamAV (file scanning) becomes its own private service, running as a Docker image, reachable only from the backend over Render's private network.

The biggest structural change is RabbitMQ: instead of standing it up as a private service, its job — dispatching and running background/executor tasks like executor's python -m backend.exec — gets replaced by Render Workflows. The rest_server's current "publish a message to a queue" calls become workflows.start_task(...) calls, and the executor's message-consuming loop becomes a @app.task-decorated Python function that Render triggers, scales, and retries automatically, removing the need for a broker instance entirely.

All of these services would sit together in one render.yaml blueprint, wired up via fromService / fromDatabase env var references so they can find each other over Render's private network without hardcoded hostnames.
